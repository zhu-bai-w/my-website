这篇文章记录一次小型个人效率系统实验：我把 Codex 当成计划助手，把滴答清单当成任务与日历数据库，再通过 MCP 把“自然语言规划”变成可以直接落地的待办事项。

它不是一个完整产品，而是一套可以继续迭代的个人工作流：先让 AI 帮我理解目标、拆分任务、判断优先级，再把整理后的结果写进我每天都会打开的计划工具里。

## 背景：为什么要把 AI 接到计划工具里

滴答清单一直是我管理日程和计划的主要工具。它对中国地区的支持比较好，也能导入其他日历；再配合 WakeUp 课程表的课程导出功能，我可以在滴答清单的日历视图里同时完成日程管理和计划安排。

这套流程已经能满足日常使用，但用久之后仍然有两个明显问题：

- 我经常会提前规划未来一段时间的任务，但每个事项都要手动分类、设定日期、安排优先级，重复操作很多。
- 我需要对一段时间内的日程、待办、项目负载有整体感知，但滴答清单更擅长承载任务，不擅长主动给出结构化分析。

2026 年，用 AI 做计划已经很常见。但从“AI 帮我想清楚”到“任务真的进入时间管理软件”，中间仍然有一道壁垒。前几天我在使用 Codex 时，看到 Notion 官方 MCP 的接入方式，于是顺手搜索了一下，发现滴答清单也提供了 MCP 能力，而且刚好覆盖我的需求。

于是这个小项目的目标就很明确了：用 Codex 搭建一个能读懂自然语言、拆分计划、给出写入建议，并最终把事项写入滴答清单的个人计划助手。

## 整体思路

我把这个助手拆成两层：

| 层级 | 承担角色 | 主要能力 |
| --- | --- | --- |
| Codex Agent | 计划助手与分析器 | 理解输入、拆分目标、分类任务、生成计划、提出写入建议 |
| 滴答清单 | 日历与任务数据库 | 存储任务、展示日程、同步多端、承载执行过程 |

Codex 负责“想清楚”：它读取我的自然语言输入，识别里面的目标、约束、时间点、任务粒度和优先级。滴答清单负责“放进去并提醒我”：它承载日历、任务清单、看板视图和跨设备同步。

这样拆分之后，AI 不需要替代任务管理软件，滴答清单也不需要变成一个分析工具。两者各自做自己擅长的部分，中间用 MCP 连接。

## Codex Agent 负责什么

在这个系统里，Codex 的核心职责不是“替我努力”，而是把模糊输入整理成可执行计划。

它主要承担五类工作：

- 读取我的自然语言输入，比如一段聊天记录、一组目标、一个项目想法，或者一张包含待办的图片。
- 拆分目标，把长期目标拆成项目，把项目拆成阶段，把阶段拆成任务。
- 分类任务，判断事项应该进入 Inbox、某个项目清单、日程安排，还是只作为想法保留。
- 生成计划，给出今天、本周或某个项目周期内的推进顺序。
- 提出写入建议，在真正写入滴答清单前，让我确认任务标题、截止时间、优先级和归属清单。

我希望它首先像一个计划助手：帮我规划长期计划，或者从聊天记录和图片里检索出待办事项。然后借助 MCP，把已经整理过的事项直接写入滴答清单。最后，它还能反向检索和分析我的日程与事项，给出建议，或者帮助我做结构化复盘。

## 滴答清单负责什么

滴答清单在这套流程中承担的是“执行系统”的角色。它有几个优势：

- 多平台覆盖比较完整，安卓、iOS、Windows、macOS 都能使用。
- 日历视图和任务清单天然适合承载日程与待办。
- 移动端提醒和桌面端整理可以配合使用。
- 它已经是我原有工作流的一部分，不需要再额外迁移数据。

这点很重要。个人效率系统最容易失败的地方，不是工具功能不够，而是新工具没有进入日常使用路径。滴答清单本来就在我的日常流程里，所以 Codex 只需要增强它，而不是替换它。

## 四个模式：把助手拆成可复用 Skill

我把这个计划助手先设计成四个模式。它们不一定要一次性全部自动化，但每个模式都对应一个真实场景。

### 1. Inbox Capture：收集模式

收集模式专门处理混乱输入。

我会把下面这些内容直接扔进去：

- 一段微信聊天记录
- 一堆突然想到的待办
- 一个项目脑暴清单
- 多个目标混在一起的碎片文本
- 图片、截图或临时备忘

它需要完成的不是简单转写，而是判断这些信息该如何进入系统：

- 提取真正的待办事项。
- 去除重复事项。
- 判断哪些只是想法，哪些已经可以执行。
- 按项目、场景或优先级分类。
- 决定哪些进入 Inbox，哪些进入具体项目清单。

这个模式解决的是“脑子里一堆想法，但整理不清楚”的困境。现在我可以先把信息一次性丢给 AI，让它完成第一轮筛选和结构化。

### 2. Daily Planner：日计划模式

日计划模式适合在一天开始时使用。它更像一个小秘书，会先检查：

- 逾期任务
- 今天截止的任务
- 高优先级任务
- 已经排进日历的固定日程
- 可以顺手推进的小任务

然后它会把任务压缩成三层：

| 层级 | 含义 |
| --- | --- |
| 必须做 | 今天不做会产生明显后果 |
| 应该做 | 对当前目标有帮助，但可以调整顺序 |
| 可选 | 状态好时推进，状态差时可以延期 |

我特意让它控制输出，尽量收缩成 1 到 3 个重点任务。日计划最重要的是可执行性，而不是看起来很完整。如果一天开始时就生成十几个“重点”，那它反而会增加心理负担。

### 3. Weekly Planner：周计划模式

周计划模式比日计划更偏项目视角。

它会检索所有任务状态，然后给我一份更完整的负载建议。很多时候问题不是不想做，而是目标定得太高，或者把太多任务堆到同一天。周计划模式要解决的就是这个问题。

它最后输出的结构大致是：

- 本周主线：最重要、最能推动当前阶段的事项。
- 本周支线：可以推进但不应挤占主线的任务。
- 可延期项：暂时不值得占用本周注意力的内容。
- 时间分配建议：把任务安排到更合理的日期，而不是全部堆在周一。

### 4. Review Retro：复盘模式

复盘模式是我目前最满意的一层。

以前做复盘时，我经常很难精确追踪自己原本安排了什么、实际完成了什么，也很难对当前整体状态有准确把握。现在，Agent 可以基于滴答清单里的任务数据生成报告，我再像分析师一样阅读报告、补充判断、写自己的复盘。

更重要的是，它可以根据我的复盘模板，用自然语言向我提问。比如：

- 本周真正推进了什么？
- 哪些任务反复延期？
- 哪些目标其实不再重要？
- 下周应该减少什么，而不是增加什么？

这降低了写复盘的阻力，也让计划、执行、复盘之间形成闭环。

## 写入流程：先建议，再确认

我不希望 Agent 直接把所有识别到的内容写进滴答清单。计划系统一旦被无效任务污染，后续整理成本会非常高。

所以我的写入流程会分成三步：

1. 先分析输入，生成候选任务。
2. 给出写入建议，包括标题、清单、日期、优先级和备注。
3. 我确认之后，再通过 MCP 写入滴答清单。

这个设计看起来多一步，但实际使用中很必要。AI 适合整理和建议，人仍然需要保留最终决策权。

## 实际体验

目前最直接的收益有三个。

第一，事项整理速度明显变快。原来我需要手动判断“这句话是不是待办”“应该放哪个清单”“有没有重复”，现在可以先让 Codex 输出整理结果，我只做确认和微调。

第二，计划的粒度更稳定。人手动写计划时，很容易一会儿写成大目标，一会儿写成小任务。让 Agent 按固定规则拆解后，任务颗粒度会更接近，后续执行也更顺。

第三，复盘更容易开始。复盘最难的不是写结论，而是重新回忆一周发生了什么。现在系统可以先把任务状态、延期项和完成情况整理出来，我只需要在这个基础上做判断。

## 这套流程的关键原则

这次实验让我更确定一个判断：个人计划助手不应该追求“全自动”，而应该追求“减少整理成本”。

我会继续保留几条原则：

- AI 负责整理、拆分、建议，不直接替我决定所有事情。
- 写入任务前必须有确认环节。
- 日计划要少而明确，周计划要关注负载。
- 复盘不只看完成数量，也要看目标是否仍然值得做。
- 系统应该接入已有习惯，而不是强迫自己迁移到全新的工具。

## 未来想法

后续我想继续补三层能力。

### 长期计划拆解

我会设计一个新的 Skill，专门负责长期计划。它需要把一个较大的目标逐步拆解到项目、阶段、里程碑和具体任务，直到最后可以写入滴答清单。

长期计划的难点在于，不是所有目标都应该马上变成任务。有些目标需要先变成探索，有些目标需要先确认资源，有些目标应该被暂时冻结。

### 接入飞书归档

我希望把复盘、计划书、阶段总结这类产出接入飞书进行存档。滴答清单适合执行，飞书更适合保存结构化文档。

如果这一步打通，计划助手就不只是写入任务，还能沉淀项目记录。

### 连接其他智能体

我还想尝试把它和另外两个智能体连接起来：

- 账本助手：把时间安排、消费计划和预算复盘联系起来。
- 信息收集助手：把收集到的信息、阅读材料和项目任务联系起来。

如果这些系统能互相调用，个人工作流就会更接近一个小型操作系统：信息进入，任务生成，执行推进，复盘沉淀。

## 小结

这次搭建的核心价值，不是“让 AI 帮我列待办”，而是把待办从自然语言里稳定地提取出来，再进入一个我已经习惯使用的执行系统。

Codex 提供理解、拆分和建议；滴答清单提供日历、提醒和跨端执行；MCP 提供两者之间的连接。三者组合起来，就形成了一个足够轻量、也足够实用的个人计划助手。
